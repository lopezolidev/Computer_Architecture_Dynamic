# Computer_Architecture_Dynamic
A platform for automated grading and parametric binary generation (anti-AI) designed to evaluate core concepts of Computer Organization and Architecture through reverse engineering, stack manipulation, binary arithmetic, and runtime analysis.

# Automated Grader for Practical Labs (x86-64 Architecture)

This repository contains the core system for the dynamic generation, deployment, and automated grading of the **Short-Duration Evaluation Labs (6 Hours Max)** for the Computer Organization and Architecture course[cite: 1].

Shifting away from traditional evaluation methods based on writing C source code (which can be easily delegated to AI tools), this platform implements a methodological approach centered on **precompiled, obfuscated binary black boxes**[cite: 1]. Students must solve optimization, reverse engineering, and binary hacking challenges by interacting directly with assembly control flows and bit-level data mutations[cite: 1].

## 🔬 Design Philosophy & AI Mitigation
To ensure academic integrity and prevent direct copying or delegation to Large Language Models (LLMs), the framework is built upon two core pillars:
1. **Parametric Solutions:** Every binary is compiled dynamically, embedding a cryptographic key bound to the student's unique national ID or Student ID (C.I.)[cite: 1]. The success token (`FLAG`) is irreversible and unique per student[cite: 1].
2. **Runtime Analysis:** The challenges require active empirical analysis, live debugging sessions (GDB), and Boolean algebra applied to hexadecimal dumps[cite: 1]—areas where static LLMs historically fail to predict execution states[cite: 1].

## ⚙️ System Architecture

The project is structured into three main components:

1. **`generator/` (Challenge Generator):**
   * A Python/C engine that ingests the core challenge templates and the student's unique ID[cite: 1].
   * Compiles an isolated binary applying the corresponding cryptographic seeds and structural mutations[cite: 1].
2. **`challenges/` (Source Templates):**
   * Contains the x86-64 assembly routines and C source templates that model the logical vulnerabilities and microarchitectural time controls (such as the native `RDTSC` instruction)[cite: 1].
3. **`grader/` (Automated Correction Script):**
   * An evaluation server subsystem[cite: 1]. It processes the student's ID, computes the expected hash, and validates it against the token submitted by the student[cite: 1].

## 🚀 Environment Prerequisites (For Students)

To solve these labs, students are recommended to use a native Linux environment (preferably **Debian 12**) equipped with the following low-level tools:
* **GCC / Make** (For writing local automation scripts or exploit proofs-of-concept).
* **GDB (GNU Debugger)** with register visualization extensions (e.g., GEF or Pwndbg).
* **Hex Editors** (Command-line tools such as `xxd`, `hexedit`, or GUI equivalents).
